# 그래프DB (Graph Database)

## 한 줄 정의
데이터를 **노드(Node, 개체)**와 **엣지(Edge, 관계)**로 표현하는 데이터베이스. 복잡한 관계 구조를 직관적으로 저장하고 탐색할 수 있다.

---

## RDBMS vs 그래프DB

| 구분 | RDBMS | 그래프DB |
|------|-------|---------|
| 데이터 구조 | 테이블 (행·열) | 노드·엣지 그래프 |
| 관계 표현 | 외래키, 조인 | 엣지(Edge)로 직접 표현 |
| 관계 변경 | 테이블·컬럼 수정 필요 | 노드·엣지 추가만으로 유연하게 변경 |
| 가독성 | 복잡한 조인은 이해 어려움 | 관계가 시각적으로 직관적 |
| 적합한 데이터 | 정형, 규칙적인 구조 | 복잡하고 유동적인 관계 구조 |
| 탐색 | SQL 쿼리 | 그래프 탐색 (경로 추적) |

> "RDB에서 관계가 바뀌면 컬럼 추가, 테이블 추가, 데이터 흐름 수정... 그래프DB는 훨씬 직관적이다." — 김선영, 엔코아

---

## 그래프DB의 구성 요소

### 노드 (Node)
개체. 온톨로지의 **인스턴스**에 해당.
- 예: `삼성전자`, `2026-05-01 세금계산서`, `공급가액`

### 엣지 (Edge)
노드 사이의 관계. 온톨로지의 **Relation**에 해당.
- 예: `삼성전자 --[발행]--> 세금계산서`, `세금계산서 --[포함]--> 공급가액`

### 속성 (Property)
노드·엣지의 부가 정보.
- 예: `세금계산서.발행일 = 2026-05-01`

---

## 온톨로지와 그래프DB의 관계

```
온톨로지 설계 (개념·관계 정의)
    ↓
그래프 모델링 (클래스 → 노드 유형, 관계 → 엣지 유형)
    ↓
그래프DB 구축 (데이터를 노드·엣지로 저장)
    ↓
AI Agent / RAG 활용 (관계 기반 탐색)
```

**온톨로지가 "무엇을 저장할지"를 정의한다면, 그래프DB는 그 구조를 실제로 구현하는 도구다.**

→ VectorDB가 의미적 유사도 기반 검색에 강하다면, 그래프DB는 **관계 경로 추적**에 강하다.

---

## AI Agent 친화적인 이유

AI Agent는 데이터를 찾을 때 단순 키워드 검색이 아니라 **관계 탐색**이 필요하다.

예: "이 계약서와 연결된 거래처의 최근 세금계산서를 찾아줘"
- RDB: 여러 테이블 JOIN 필요, 쿼리 복잡
- 그래프DB: 계약서 노드 → 거래처 노드 → 세금계산서 노드 경로를 자연스럽게 탐색

> 기업 직원처럼 "프로세스, 관계, 암묵적 규칙"을 알고 있어야 AI가 제대로 일할 수 있다.
> 그래프DB가 그 '지도'를 제공한다. → [[지식-그래프-디지털-맥락지도]]

---

## EDA(탐색적 분석) 관점의 장점

그래프DB에서 데이터를 시각화하면 **생각지 못했던 관계**가 눈에 보인다.
- "이 노드와 저 노드가 이렇게 연결되어 있구나"
- 빅데이터 분석의 EDA처럼 → 예상치 못한 인사이트 발견 가능

## 실용 도구

| 도구 | 특징 |
|------|------|
| **Neo4j** | 가장 널리 쓰이는 그래프DB. 개발자 친화적 라이브러리 풍부. |
| **Amazon Neptune** | AWS 관리형 그래프DB |
| **ArangoDB** | 멀티모델 (그래프 + 문서 + 키-값) |

> Neo4j는 시작이 빠르고 유틸리티가 잘 되어 있어 가장 많이 사용된다. 단, 기능적 제약과 실시간 동기화 이슈는 고려 필요.

## VectorDB vs 그래프DB (RAG 관점)

| | VectorDB | 그래프DB |
|--|---------|---------|
| 검색 기반 | 의미적 유사도 (임베딩) | 관계 경로 탐색 |
| 강점 | "이것과 비슷한 내용 찾기" | "이것과 연결된 것 찾기" |
| 약점 | 관계 구조 표현 어려움 | 비정형 텍스트 의미 검색 약함 |
| 조합 | **GraphRAG**: 두 접근을 결합 → 더 정확한 컨텍스트 검색 |
