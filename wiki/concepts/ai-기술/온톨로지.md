# 온톨로지 (Ontology)

## 한 줄 정의
특정 도메인에서 **개념이 무엇인지, 개념들 사이 관계가 무엇인지, 어떤 조건에서 동일하고 구분되는지**를 명시한 공유된 해석 기준 체계.

> 데이터 모델이 "어디에 저장할 것인가"를 정의한다면,
> 온톨로지는 "그것이 무엇을 의미하는가"를 고정한다.

---

## 공식 정의

**Gruber (1993)**
> "an explicit and formal specification of a conceptualization of a domain of interest"
> → 관심 도메인을 개념화하기 위해 명시적으로 정형화한 명세서

**Borst**
> "a formal specification of a shared conceptualization"
> → Gruber에 **공유(shared)** 개념 추가. 그룹 구성원이 **합의**한 개념 체계.

---

## 왜 필요한가 — "board" 예시

같은 단어도 맥락에 따라 의미가 완전히 달라진다.

| 맥락 | board의 의미 |
|------|------------|
| 눈(snow)과 연결 | 스노보드 |
| 음식과 연결 | 식탁 |
| 인터넷과 연결 | 게시판 |
| 학교·학부모와 연결 | 위원회 |
| 회사와 연결 | 이사회 |

사람은 문맥으로 의미를 찾는다. **온톨로지는 이 문맥 해석을 기계가 수행할 수 있도록 명시적으로 구조화하는 것.**

## 데이터 모델 vs 온톨로지

| 구분 | 데이터 모델 | 온톨로지 |
|------|------------|---------|
| 목적 | 저장 구조 정규화 | 의미·해석 기준 정규화 |
| 질문 | "어디에, 어떤 형식으로 저장하나?" | "이것은 어떤 개념이고, 무엇과 같은가?" |
| 한계 | '공급가액'과 '공급금액'이 같은지 모름 | 두 표현을 하나의 업무 개념으로 묶음 |
| 구성 요소 | 테이블, 컬럼, 키, 관계 | 클래스, 인스턴스, 속성, 관계 |

> "데이터 모델은 저장을 정규화하지만, 의미를 정규화하지는 않는다."
> "데이터 모델이 시스템의 뼈대라면, 온톨로지는 판단의 기준선이다."

둘은 서로 다른 문제를 푼다. **데이터 모델 위에 온톨로지가 놓여야 완성된 시스템이 된다.**

---

## 온톨로지의 4가지 구성 요소

### 1. 클래스 (Class)
개념의 이름·범주. 세계에 존재하는 사물이나 개념의 집합.
- 예: `세금계산서`, `계약서`, `공급자`, `공급가액`

### 2. 인스턴스 (Instance)
클래스의 구체적 실체. 사물이나 개념이 실질적 형태로 나타난 것.
- 예: `삼성전자가 2026-05-01에 발행한 세금계산서 #1234`
- 주의: 클래스/인스턴스 구분은 사용 목적에 따라 달라질 수 있음

### 3. 속성 (Property)
클래스·인스턴스의 특성 → 특정 값(value)과 연결.
- 예: `세금계산서 hasField 공급가액`, `공급가액 hasType 금액`

### 4. 관계 (Relation)
클래스·인스턴스 사이의 연결.

| 관계 유형 | 설명 | 예시 |
|----------|------|------|
| **Taxonomic (isA)** | 계층적 포함 관계 | `세금계산서 isA 기업문서` |
| **Non-taxonomic** | 인과, 포함, 연결 등 | `공급가액 cause 세액계산` / `계약서 hasPart 특약조항` |

---

## 온톨로지의 유형

### 구축 범위에 따른 분류

| 유형 | 설명 | AX 관련성 |
|------|------|----------|
| **일반 온톨로지** | 공간, 시간, 사건 등 보편적 개념 | 낮음 |
| **영역 온톨로지 (Domain Ontology)** | 특정 도메인의 개념 체계 | **★★★ 가장 높음** |

**영역 온톨로지가 Dohun의 AX 목표와 직결:**
- 선박 도메인: BL, 선적항, 양하항, 수화인 등의 개념 체계
- 제조 도메인: 품목, 불량 유형, 검사 기준 등의 개념 체계
- 금융 도메인: 세금계산서, 공급가액, 세액 등의 개념 체계

---

## 온톨로지 구현 언어

| 언어 | 특징 | 용도 |
|------|------|------|
| **RDF** | subject-predicate-object 트리플 구조. 단순 관계 표현. | 기본 개념 연결 |
| **OWL** | 관계 계층 + 논리적 제약조건 + 추론 가능. 정밀한 온톨로지. | 복잡한 도메인 온톨로지 구현 |
| **SWRL** | 추론 규칙 정의. | 규칙 기반 자동 추론 |

> 실무 AX 온톨로지를 구현하려면 **OWL**이 가장 표준적인 선택.
> 그래프 DB(Neo4j)를 활용하는 실용적 접근도 있음.

## 온톨로지의 3가지 역할

### 1. 개념 정의 (Concept Definition)
도메인 안에 어떤 개념이 존재하는지 명시.
- 예: "세금계산서"는 공급자, 공급받는자, 공급가액, 세액, 작성일로 구성된다.

### 2. 관계 정의 (Relation Definition)
개념들 사이의 관계를 명시.
- 예: "세금계산서" is-a "기업 문서" / "공급가액" part-of "세금계산서"

### 3. 해석 기준 고정 (Semantic Normalization)
다양한 표현을 하나의 업무 개념으로 수렴.
- "공급가액" = "공급금액" = "금액(부가세 제외)" → 동일 개념으로 처리
- **이것이 RAG와 AI 에이전트에서 가장 중요한 역할**

## RAG 파이프라인에서의 온톨로지

### 온톨로지 없을 때
```
[사용자 질문] → [벡터 유사도 검색] → [확률적 문서 선택] → [확률적 해석] → [답변]
```
- 같은 개념이 다른 표현으로 등장 → 검색 결과가 매번 달라짐
- 결과: **할루시네이션**, 일관성 없는 답변

### 온톨로지 있을 때
```
[사용자 질문] → [온톨로지 매핑] → [개념 기반 검색] → [기준 내 해석] → [일관된 답변]
```
- "공급금액" 입력 → 온톨로지에서 "공급가액" 동의어로 매핑 → 모든 관련 표현 포함 검색
- 오탈자 교정 + 동의어 확장 + 관련어 연결까지 가능

## Dohun의 핵심 인사이트

> **"청킹 전략을 고민하는 것 자체가 비효율일 수 있다. 데이터를 처음부터 온톨로지로 설계했다면, 많은 RAG 엔지니어링 문제가 줄어들었을 것이다."**

VectorDB 구축 시 표(Table)가 청킹 중에 잘리는 문제를 겪고 청킹 전략을 수동으로 조절했다. 근본 원인: "이 표가 어디에서 시작하고 끝나는가"(= 의미 경계)가 정의되지 않았기 때문.

온톨로지가 있었다면 → "이 표는 '세금계산서의 품목 목록' 개념이다" → 의미 경계가 미리 정의 → 표 잘림 문제가 구조적으로 해결.

### 업스트림 설계가 다운스트림 엔지니어링을 결정한다

```
온톨로지 설계 (업스트림)
    ↓
데이터 수집 · 파싱
    ↓
청킹 (의미 단위로 자동 결정)   ← 고민이 줄어드는 지점
    ↓
임베딩 · VectorDB 저장
    ↓
검색 (개념 기반으로 정확도 향상) ← 고민이 줄어드는 지점
    ↓
생성 (해석 기준 내에서 일관된 답변)
```

## 기업 온톨로지 구축 단계 (AX 실무)

1. **도메인 문서 수집** — 해당 산업의 핵심 문서 유형 파악
2. **개념(클래스) 추출** — 반복 등장하는 핵심 개념 (KIE 활용 가능)
3. **관계 정의** — isA, part-of, related-to, cause 등
4. **동의어 매핑** — 같은 개념의 다양한 표현 목록화
5. **속성 정의** — 각 클래스가 가지는 속성 값 정의
6. **인스턴스 연결** — 실제 데이터를 클래스에 매핑
7. **검증** — 실제 문서에 온톨로지 적용 → 일관성 확인
8. **RAG 파이프라인 연동** — 검색·생성 단계에 온톨로지 주입

## 온톨로지의 구현 도구: 그래프DB

온톨로지를 설계했다면, 그것을 저장하고 탐색할 수 있는 도구가 필요하다.

| 저장 방식 | 특징 | 온톨로지 표현력 |
|----------|------|--------------|
| RDBMS | 정형 구조, 관계 변경 어려움 | 낮음 |
| **그래프DB** | 노드·엣지로 관계 직관적 표현, 유연성 높음 | **높음** |
| VectorDB | 의미적 유사도 검색 | 관계 구조 표현 약함 |

**온톨로지 설계 → 그래프 모델링 → 그래프DB 구축** 이 자연스러운 구현 경로다.

→ 자세히: [[그래프DB]], [[지식-그래프-디지털-맥락지도]]
