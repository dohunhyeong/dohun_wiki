## 왜 저장했나
RAG 파이프라인을 직접 만들면서 겪었던 문제와 연결. 특히 청킹 전략(표 잘림 등)을 조절하는 데 많은 시간을 썼는데, **"처음부터 온톨로지로 데이터를 설계했다면 이 고민 자체가 줄어들지 않았을까"** 라는 질문에서 출발.

---

## 핵심 주장 (Key Claims)

1. **할루시네이션 = 모델 문제가 아니라 '의미 기준' 부재의 문제**
   - AI가 틀리는 근본 원인은 해석 기준이 정의되지 않았기 때문

2. **데이터 모델 ≠ 온톨로지 — 둘은 정규화 대상이 다르다**
   - 데이터 모델: 저장 구조 정규화 ("어디에 저장할 것인가")
   - 온톨로지: 의미 정규화 ("어떤 개념인가, 어떻게 해석할 것인가")
   - "데이터 모델은 저장을 정규화하지만, 의미를 정규화하지는 않는다."

3. **RAG에서 온톨로지 없이는 확률적 해석이 발생**
   - "공급가액" = "공급금액" = "금액(부가세 제외)" → 온톨로지 없이는 AI가 같은 개념인지 확률적으로 판단

4. **AI 에이전트는 의미 오류를 허용하지 않는다**
   - 단순 답변 생성이 아닌 조건 판단·시스템 입력 → 의미 오류 = 업무 오류

> **핵심 문장**: "데이터 모델이 시스템의 뼈대라면, 온톨로지는 판단의 기준선이다."

## 연결되는 나의 경험

### 청킹 전략과 온톨로지의 연결
RAG 파이프라인 구축 시 표(Table)가 잘리는 청킹 문제를 겪었고 전략을 조절했다. 이 글을 읽으며 든 인사이트:

> **"청킹 전략을 고민하는 것 자체가 비효율일 수 있다. 데이터를 처음부터 온톨로지로 설계했다면, '이 표는 무엇을 의미하는가'가 구조 안에 이미 정의되어 있었을 것이다."**

표 잘림 문제는 결국 "표의 의미 경계"를 모르기 때문에 발생하는 문제 → 온톨로지가 그 의미 경계를 미리 정의한다.

### 앞으로 해보고 싶은 것
1. 온톨로지를 RAG 파이프라인에 직접 적용 (의미 기반 검색)
2. AX 프로젝트에서 기업 데이터를 수집하고 도메인 온톨로지 구축
