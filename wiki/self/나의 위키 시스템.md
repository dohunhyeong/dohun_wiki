# 나의 위키 시스템

내가 어떻게 지식을 모으고, 저장하고, 축적하는지를 기록한 페이지.
시스템이 바뀔 때마다 여기를 업데이트한다.

---

## 전체 구조

```
웹 → [Obsidian Web Clipper] → raw/  →  [AI 인제스트] → wiki/  →  output/
     (소스 유형별 자동 분류)   (불변 원본)  (요약·연결·정제)   (보고서·발표)
```

---

## 인풋 레이어 — Obsidian Web Clipper

브라우저에서 웹 페이지를 클리핑할 때 **소스 유형에 따라 다른 템플릿**이 자동으로 적용된다.
클리퍼 파일은 볼트 루트(`llm_wiki/*.json`)에 보관한다.

### 클리퍼 목록

| 클리퍼 | raw_type | 저장 경로 | 자동 적용 사이트 |
|--------|----------|-----------|-----------------|
| 강의 노트 | `lecture` | `raw/lectures` | YouTube, Coursera, Udemy, edX, fast.ai, d2l.ai, kmooc.kr, class101.net, inflearn.com |
| 아티클 / 블로그 | `article` | `raw/articles` | Wikipedia (수동 포함) |
| 논문 (Paper) | `paper` | `raw/papers` | arXiv, Semantic Scholar, PubMed, ACM, IEEE, OpenReview, ACL Anthology, Springer, Nature, ScienceDirect, SSRN |
| 중요 대화 메모 | `conversation` | `raw/conversations` | Claude.ai, ChatGPT, Gemini, Perplexity, Poe |
| 책 챕터 노트 | `book` | `raw/books` | 수동 (트리거 없음) |
| AX 프로젝트 문서 | `project` | `raw/projects` | Notion, Google Docs/Slides, Confluence, ClickUp, Linear, Miro |
| 기본 템플릿 | — | `Clippings` | 수동 폴백 |

### 공통 설계 원칙

- **`ingest_status: pending`**: 클리핑 직후 AI 인제스트 대기 상태를 표시하는 공통 프로퍼티
- **인제스트 메모 섹션**: 각 클리퍼마다 소스 유형에 맞는 질문이 본문 하단에 삽입됨

  | 클리퍼 | 인제스트 메모 항목 |
  |--------|-------------------|
  | 강의 | 핵심 메시지, 이해 안 된 부분, 연결 개념, 다음 세션 |
  | 논문 | 핵심 기여, 연결 개념, 실용적 적용점, 한계·주의사항 |
  | 대화 | 얻은 인사이트, 나의 생각 변화, 연결 개념·사람, 기록 이유 |
  | 책 챕터 | 핵심 주장, 기억할 예시, 연결 개념, 다음 챕터 |
  | AX 프로젝트 | 문서의 의미, 결정/미결 사항, 연결 위키 페이지, 후속 액션 |
  | 아티클 | 핵심 포인트, 연결 개념, 왜 저장했나 |

---

## 처리 레이어 — AI 인제스트 워크플로우

`ingest_status: pending` 인 raw/ 파일을 발견하면:

1. 소스 읽기 → 핵심 포인트 파악
2. Dohun과 주요 인사이트 확인 (방향 조율)
3. `wiki/` 에 요약 페이지 생성
4. 관련 개념·프로젝트·사람 페이지 업데이트
5. `wiki/index.md` 업데이트
6. `wiki/log.md` 에 인제스트 기록 추가

---

## 설계 배경

RAG 기반 챗봇을 직접 개발하면서 **지식 구조의 중요성**을 체감했다.
소스가 아무리 많아도 구조 없이 쌓이면 검색도 연결도 안 된다는 걸 몸으로 배움.
그래서 인풋 단계부터 소스 유형을 분류하고, AI가 일관된 방식으로 정제할 수 있도록 설계했다.
